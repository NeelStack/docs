# NES-705 — Analytics

> **"Product metrics trace user adoption. We design structured event schemas, reconstruct sessions, and analyze user journeys using standardized queries."**

---

# Executive Summary

To improve our applications, validate features, and measure user engagement, we must capture user interaction metrics.

Ad-hoc event tracking without unified schemas leads to inconsistent metrics, high noise, and unusable product dashboards.

We mandate the centralization of all behavioral telemetry under a unified **Product Analytics Pipeline**.

This standard establishes the event ingestion pipeline, session reconstruction rules, and user-funnel calculations.

---

# Purpose

This standard defines:

- Product Analytics Pipeline (Kafka to Data Lake)
- Event Naming Schemes & Validation
- Session Reconstruction Algorithms
- User Funnel Analytics Calculations
- Privacy Filters for Analytics Data

---

# Product Ingestion Pipeline

Behavioral event logs generated by web apps (NES-313) or mobile apps (NES-413) follow a decoupled ingestion flow:

```text
 Client (Web/Mobile SDK) ──► HTTP Gateway ──► Kafka Ingest Topic
                                                    │
                                                    ▼
                                           Vector Event Consumer
                                                    │
                                                    ▼
                                            S3 Parquet Lake
                                                    │
                                                    ▼
                                             dbt aggregations
```

# Unified Event Schema

To ensure that analytical query scripts compile across all products, events must conform to a standardized JSON schema:

```json
{
  "event_id": "evt_d83a74e982",
  "event_name": "user_completed_onboarding",
  "app_name": "eduos-mobile",
  "session_id": "sess_1234567890",
  "user_id": "usr_9999",
  "tenant_id": "tenant_12",
  "timestamp": "2026-07-04T12:00:00.000Z",
  "properties": {
    "step_count": 5,
    "completed_duration_sec": 120
  }
}
```

- **Object-Action Naming**: Use lowercase snake_case for event names (e.g. `document_downloaded`, `session_started`).

---

# Session Reconstruction (Sessionization)

Raw events are flat. To calculate engagement time or trace journeys, events must be grouped into **Sessions**.

- **Session Definition**: A session is a sequence of events from a single user ID where the time difference between consecutive events is less than **30 minutes**.
- **Reconciliation Job**: Run nightly dbt sessionization models to map events to session IDs, calculating total duration, session depth, and landing page URLs.

---

# User Funnel & Cohort Calculations

Standardize funnel analysis queries:

- **Funnel Calculation**: Evaluate conversion rates between sequential steps (e.g. `Landing Page` -> `Sign Up` -> `First Upload`).
- **Cohort Analysis**: Group users based on their registration week/month and trace their weekly retention rates over a 90-day window to evaluate product stickiness.

---

# Privacy Controls

Analytics data is subject to privacy laws (GDPR/HIPAA):

- **PII Filter**: Do not log raw passwords, credit cards, or names in properties.
- **Anonymization**: Encrypt user IDs in analytics stores to prevent direct user identification by default database query roles.

---

# Anti-Patterns

❌ **Tracking Button Elements directly**: Logging events like `clicked_submit_green_button` instead of modeling the semantic action (e.g. `invoice_submitted`).

❌ **Ignoring Session Timeouts**: Running session calculations that merge events across days without applying the 30-minute inactivity boundary limit.

❌ **Exposing Telemetry Keys**: Hardcoding administrator write tokens inside public web repositories.

---

# Production Checklist

- [ ] Telemetry gateways use TLS 1.3 encryption.
- [ ] Schema validation checks run on ingestion.
- [ ] Sessionization dbt jobs run nightly.
- [ ] User ID anonymization is active in analytics tables.
- [ ] Privacy controls conform to GDPR and HIPAA constraints.

---

# Success Criteria

The Analytics program is successful when:
- Product managers can generate conversion funnels without manual data cleanup.
- Event ingestion latencies from client tap to S3 remain under 5 minutes.
- Analytical reports show zero leaked PII.
