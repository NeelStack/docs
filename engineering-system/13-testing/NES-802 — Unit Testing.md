# NES-802 — Unit Testing

> **"Code modules must prove correctness in isolation. We write deterministic, fast unit tests with clear mocking boundaries."**

---

# Executive Summary

Unit tests are the foundation of our testing pyramid (NES-800).

If unit tests are slow, non-deterministic (flaky), or depend on external networks and databases, developers will stop running them locally, leading to code regression.

This standard establishes our unit testing guidelines, approved frameworks, mocking boundaries, and coverage metrics.

---

# Purpose

This standard defines:

- Standardized Testing Frameworks (Pytest, Vitest)
- Test Independence and Determinism
- Mocking and Stubbing Guidelines
- Code Coverage Rules (Target: 80%)
- CI Pipeline Automation Settings

---

# Standardized Testing Frameworks

All applications must utilize the officially approved testing frameworks:

| Runtime Stack | Primary Test Runner | Key Assertion Libraries |
|---|---|---|
| Python (Backend) | **pytest** | `pytest-cov`, `pytest-mock` |
| React / TypeScript (Web) | **Vitest** | `React Testing Library`, `happy-dom` |
| React Native (Mobile) | **Jest** | `@testing-library/react-native` |

---

# Test Independence & Determinism

To prevent flakiness and ensure speed:

- **Isolated State**: Unit tests must not depend on the execution order or outcomes of other tests. Each run must initialize and tear down its own mock memory state.
- **Zero Network/Database Calls**: Unit tests must never initiate real network requests (HTTP) or execute local database queries. Mock these calls entirely.
- **Deterministic Rules**: Avoid writing tests that rely on dynamic system states (e.g. system timezone, current date) unless mocked.

---

# Mocking & Stubbing Standards

Mock external dependency interfaces to keep test runs fast:

- **Mock Boundaries**: Mock external API clients, cloud object stores, and database models.
- **Strict Mocks**: Configure mocks to assert that they are called with the expected arguments:

```python
# Reference Pytest Mocking using pytest-mock
def test_student_onboarding_sends_email(mocker):
    # Mock the external email provider client interface
    mock_mailer = mocker.patch("services.email.Mailer.send_welcome_email")
    
    # Execute application logic
    onboard_new_student(email="test@neelstack.com")
    
    # Assert dependency was called correctly in isolation
    mock_mailer.assert_called_once_with("test@neelstack.com")
```

# Coverage Configurations & CI Gates

Ensure testing quality is maintained during updates:

- **Config File**: Maintain coverage exclusion parameters in config files (e.g., `vitest.config.ts`, `pyproject.toml`) to skip autogenerated database schemas, UI boilerplate, or migration files.
- **Coverage Gate**: CI pipelines must enforce the **80% line coverage** requirement on all modified files in pull requests.

---

# Anti-Patterns

❌ **Testing Core Framework Code**: Writing unit tests to verify that built-in framework features (e.g. FastAPI routing, Next.js page transitions) work instead of testing custom business logic.

❌ **Over-Mocking internal methods**: Mocking internal private functions of the module under test, which makes tests brittle and prone to breaking when code is refactored. Mock outer interfaces only.

❌ **Leaving Assert-less Tests**: Writing tests that execute code paths but lack `assert` checks, which verify execution path health but fail to validate output correctness.

---

# Production Checklist

- [ ] Approved runners (pytest/vitest) are configured.
- [ ] Unit tests contain zero external socket connections.
- [ ] Code coverage thresholds are active in CI configurations.
- [ ] Mocks assert arguments and calling metrics.
- [ ] Test execution logs are recorded in HTML reporting databases.

---

# Success Criteria

The Unit Testing program is successful when:
- Unit test suites run and complete in less than 2 minutes.
- Code refactoring can be completed safely without changing unit test structures.
- Bug regressions decrease as unit test coverage expands.
